# 技术问答提示词测试方案

## 1. 测试目的

评估“技术速答类提示词”是否能让 LLM 在回答软件工程问题时实现以下目标：

- 回答更短
- 结论更靠前
- 重点更突出
- 保持准确性
- 保持工程实用性
- 输出风格稳定

适用问题范围包括：

- 前端
- 后端
- 数据库
- 微服务 / 分布式
- DevOps
- Agent / RAG / LLM 工程

---

## 2. 测试对象

### 提示词版本

- **A 版本（基线版）**
  - 当前常用提示词 / 无提示词 / 旧提示词

- **B 版本（实验版）**
  - 新设计的“短答、重点突出”提示词

> 建议使用同一批问题同时测试 A/B 两个版本，做对比评估。

---

## 3. 测试方法

### 3.1 A/B 对比测试

对同一个问题分别使用：

- A 提示词
- B 提示词

对比以下维度：

- 字数
- 是否结论前置
- 是否重点突出
- 是否准确
- 是否具备工程实用性

---

### 3.2 稳定性测试

建议每个问题每个版本执行 **3 次**，观察：

- 输出长度是否波动过大
- 输出结构是否一致
- 是否容易“失控变长”

---

### 3.3 真实场景补充测试

除标准题库外，建议补充你最近真实问过的技术问题，以验证提示词在真实工作流中的效果。

---

## 4. 评估维度

每题建议按 1~5 分评分。

| 维度 | 说明 |
|---|---|
| 简洁度 | 是否足够短，没有明显废话 |
| 结论前置 | 是否优先给出核心结论 |
| 重点突出 | 是否聚焦最关键的信息 |
| 准确性 | 内容是否正确，无明显错误 |
| 实用性 | 是否对工程实践有帮助 |
| 稳定性 | 多次输出风格和质量是否一致 |

---

## 5. 量化指标

除主观评分外，建议额外记录以下客观指标：

| 指标 | 说明 |
|---|---|
| 回答字数 | 用于衡量篇幅控制能力 |
| 结论前置 | 是否在开头给出结论（Y/N） |
| 要点数量 | 是否控制在期望范围内 |
| 示例数量 | 是否提供最小必要示例 |
| 无关展开 | 是否出现明显跑题或冗长铺垫 |
| 版本/前提提醒 | 是否在必要时提醒版本差异或适用前提 |

---

## 6. 通过标准

若目标是“更短、更聚焦、更适合日常技术速答”，建议通过标准如下：

| 指标 | 目标 |
|---|---|
| 平均字数 | 相比基线下降 30% 以上 |
| 结论前置命中率 | 80% 以上 |
| 重点突出平均分 | 4 分以上 |
| 准确性 | 不低于基线版本 |
| 稳定性 | 同类问题 3 次回答风格基本一致 |

---

## 7. 标准测试用例

> 说明：可先使用以下标准题库进行 A/B 测试，再补充真实业务问题。

| 用例ID | 类型 | 测试问题 | 预期观察点 |
|---|---|---|---|
| C1 | 概念型 | Java 里的 volatile 关键字有什么作用？ | 是否先说明“保证可见性，不保证复合操作原子性”；是否避免长篇展开 JVM 内存模型；是否短而准 |
| C2 | 概念型 | HTTP 301 和 302 的核心区别是什么？ | 是否快速指出“永久重定向 vs 临时重定向”；是否提到缓存 / SEO / 方法语义；是否不冗长 |
| D1 | 对比型 | Redis String 和 Hash 在什么场景下怎么选？ | 是否直接给选型建议；是否用少量要点讲清差异；是否避免泛泛而谈 |
| D2 | 对比型 | 乐观锁和悲观锁的主要区别是什么？ | 是否抓住“冲突处理方式”；是否说明适用场景；是否短而有效 |
| P1 | 实践型 | Spring Boot 项目里，接口幂等一般怎么做？ | 是否优先给出常见工程方案；是否偏实践；是否不过度扩展 |
| P2 | 实践型 | MySQL 分页为什么深分页慢？一般怎么优化？ | 是否指出 offset 扫描成本；是否给出 keyset pagination / 延迟关联；是否重点明确 |
| T1 | 排障型 | 线上接口突然变慢，应该先排查什么？ | 是否给出有顺序的排查路径；是否覆盖应用、数据库、依赖、资源；是否短而不空 |
| T2 | 排障型 | Kafka 消费堆积，通常先看哪些指标？ | 是否直接列关键指标；是否有优先级；是否避免长篇原理说明 |
| K1 | 代码型 | 用 Python 写一个 LRU 缓存，尽量简单。 | 是否给最小可用实现；是否代码简洁；是否不过度解释 |
| K2 | 代码型 | 给我一个 Go 里带超时控制的 HTTP 请求示例。 | 是否代码可直接使用；是否足够短；是否解释 timeout 的关键位置 |
| A1 | Agent / RAG | RAG 里的 rerank 主要是干什么的？ | 是否一句话说明本质；是否提到“召回后重排序”；是否不过度扩展 |
| A2 | Agent / RAG | Agent 和传统工作流的核心区别是什么？ | 是否抓住“自主决策 vs 固定流程”；是否清晰简短 |
| G1 | 干扰测试 | Redis 为什么快？ | 是否覆盖核心原因；是否短答；是否不会自动展开成长篇 |
| G2 | 干扰测试 | 什么是微服务？ | 是否保持速答模式；是否避免教科书式长定义 |
| F1-1 | 追问测试 | Kafka 为什么会重复消费？ | 第一轮是否短答；是否只回答核心原因 |
| F1-2 | 追问测试 | 展开讲讲 offset 提交机制。 | 第二轮是否按需展开；是否体现“默认短答、追问再展开” |

## 8. 执行步骤

### 第一步：基础 A/B 测试
- 使用标准测试用例
- 每题分别测试 A / B 两个提示词版本
- 记录输出和评分

### 第二步：稳定性测试
- 从标准题库中选 5 题
- 每题每个版本执行 3 次
- 比较字数、结构、重点是否稳定

### 第三步：真实问题测试
- 加入最近真实工作中问过的 5~10 个问题
- 验证提示词在日常使用中的适配度

## 12. 最终建议

如果你的目标是：

> 让 LLM 在日常技术小知识点问答中默认输出“短、准、聚焦”的回答

那么测试时最应重点关注这 4 项：

1. **字数是否明显下降**
2. **结论是否前置**
3. **重点是否集中**
4. **准确性是否没有明显下降**

核心原则不是“越短越好”，而是：

> **在不明显牺牲准确性和实用性的前提下，尽量缩短回答并突出重点。**
